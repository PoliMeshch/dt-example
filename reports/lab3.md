### Этап 1. Для начала

#### 1.1. Используемые программы

На постоянной основе используются следующие программы:

- **редактирование текстовых файлов:** Visual Studio Code;
- **программирование на основном языке (C++):** Visual Studio Code;
- **выполнение команд в консоли:** Git Bash, терминал WSL (через Windows Terminal или VS Code);
- **навигация по файловой системе:** проводник Windows, командная строка, Visual Studio Code;
- **работа с Git:** Git Bash, Visual Studio Code.

#### 1.2. Альтернативы

**Текстовые редакторы:**
- **Sublime Text** — быстрый и легкий, но нет встроенной интеграции с Git;
- **Neovim** — мощный, полностью управляется с клавиатуры, но требует настройки и привыкания.

**Среды разработки:**
- **Visual Studio** — тяжелая IDE, но с мощным отладчиком и компилятором "из коробки".

**Консоль:**
- **PowerShell** — более мощный, но синтаксис команд отличается от Unix.

**Git-клиенты:**
- **GitHub Desktop** — удобный графический интерфейс для базовых операций, но не подходит для 
сложных сценариев.

**Источники:** информация собрана с официальных сайтов программ:

#### 1.3. Опыт использования альтернатив

Для знакомства с альтернативами были выбраны **Neovim** (как пример мощного консольного редактора) 
и **GitHub Desktop** (как пример графического Git-клиента).

**Neovim:** установила через `winget`, настроила базовый конфиг (номера строк, подсветка). В течение 
нескольких дней пользовалась им вместо VS Code — открывала лабораторные файлы, правила комментарии, 
добавляла пустые строки.

Основное, что мешало: режимы (normal/insert) — постоянно забывала переключиться, начинала печатать, 
а текст не появлялся; отсутствие автодополнения и подсказок по ошибкам (для C++ нужно отдельно 
настраивать LSP-сервер, я не стала). Несколько раз случайно нажимала `:wq` вместо `:q`, когда не 
хотела сохранять изменения, — приходилось потом отменять лишние правки.

**Вывод:** для моих задач (лабораторные по структурам данных, где нужно быстро писать и отлаживать код) 
VS Code оказался удобнее и понятнее. Neovim оставлю для случаев, когда нужно быстро что-то поправить на 
сервере через SSH — там он действительно удобен.

**GitHub Desktop:** установила через `winget install github-desktop`, авторизовалась в аккаунте. 
Клонировала свой репозиторий `dt-example` в отдельную папку.

Открыла README.md через встроенный редактор, добавила тестовую строку, сделала коммит и нажала Push — 
изменения отправились на GitHub, я проверила в браузере, всё появилось. Ещё посмотрела вкладку History: 
коммиты отображаются с сообщениями, датами и хэшами — это нагляднее, чем `git log` в терминале.

Основное, что не понравилось: интерфейс медленнее, чем работа в командной строке, и нет привычных команд 
(`rebase`, `reset`). Для простых операций (коммит, пуш) подходит, но для того, что я делаю в лабораторных, 
Git Bash удобнее.

**Вывод:** GitHub Desktop можно использовать для быстрых коммитов и пуша, но полностью переходить на него 
не планирую.

#### 2.3. Настройка приглашения ко вводу

**Что такое PS1**

PS1 (Prompt String 1) — переменная окружения в bash, задающая формат приглашения командной 
строки. Позволяет отображать полезную информацию: имя пользователя, текущую папку, ветку Git и др.

**Настройка**

Для настройки приглашения в WSL отредактирован файл `~/.bashrc`:

```bash
nano ~/.bashrc
```

В конец файла добавлена строка:

```bash
PS1='\[\e[32m\]\u@\h \[\e[34m\]\W \[\e[33m\]$(__git_ps1 "(%s)")\[\e[0m\]\$ '
```

**Значение элементов:**

| Элемент | Значение |
|---------|----------|
| `\u` | имя пользователя |
| `\h` | имя хоста |
| `\W` | текущая директория (без полного пути) |
| `$(__git_ps1 "(%s)")` | отображение текущей ветки Git |
| `\[\e[32m\]` | зелёный цвет |
| `\[\e[34m\]` | синий цвет |
| `\[\e[33m\]` | жёлтый цвет |
| `\[\e[0m\]` | сброс цвета |
| `\$` | символ приглашения |

**Применение настроек:**

```bash
source ~/.bashrc
```

**Результат:** После выполнения команды приглашение изменилось. Теперь оно отображает имя 
пользователя, хост, текущую папку и ветку Git. Это удобно, так 
как сразу видно, в какой ветке происходит работа.

*Теперь приглашение выглядит так:*

```
darel@MAIBENBEN-PC dt-example (lab3-report)$
```

*Имя пользователя и хост зелёные, текущая папка синяя, ветка Git жёлтая.*

- [Git in PS1](https://git-scm.com/docs/git-prompt)


### Этап 3. Эргономика работы с кодом

#### 3.1. Горячие клавиши VS Code

Список горячих клавиш можно открыть через *File → Preferences → Keyboard Shortcuts* или 
сочетанием `Ctrl+K` `Ctrl+S`.

**Наиболее интересные горячие клавиши:**

- `Ctrl+Enter` — вставить строку ниже и перейти на неё
- `Ctrl+Shift+Enter` — вставить строку выше
- `Ctrl+Shift+K` — удалить всю строку
- `Alt+↑` / `Alt+↓` — переместить строку вверх/вниз
- `Shift+Alt+↑` / `Shift+Alt+↓` — копировать строку вверх/вниз
- `Alt+Click` — множественный курсор
- `Ctrl+H` — найти и заменить
- `Ctrl+P` — поиск файла по имени
- `Ctrl+T` — поиск символа (функции, класса)

#### 3.2. Полезные функции редактора

- **Поиск по содержанию файлов** (`Ctrl+Shift+F`) — открывает боковую панель поиска по всему проекту.
- **Перейти к объявлению функции или типа** (`F12` или `Ctrl+Click`) — открывает файл с объявлением.
- **Перейти к реализации** (`Ctrl+F12`) — переходит к реализации функции (например, из `.hpp` в `.cpp`).
- **Найти все места использования** (`Shift+F12`) — показывает список всех ссылок на символ.
- **Переименовать все вхождения** (`F2`) — переименовывает символ во всех местах использования.
- **Быстро перемещаться между интересными местами кода** (`Alt+Left` / `Alt+Right`) — навигация по истории курсора.
- **Комментировать / раскомментировать блок** (`Ctrl+/`) — добавляет или убирает `//` в начале выделенных строк.

**Примечание:** В VS Code также есть панель *Go to Symbol* (`Ctrl+Shift+O`), которая позволяет быстро 
перейти к любому символу в текущем файле.

### Этап 5. Интеграция с Git

В VS Code есть встроенная интеграция с Git. Работать с репозиторием можно как через терминал, так и через 
панель *Source Control*, которая открывается значком слева или сочетанием `Ctrl+Shift+G`.

- **Просмотр изменений** — в разделе *Source Control → Changes* отображаются изменённые файлы. При нажатии 
на файл открывается сравнение с последней версией.
- **Добавление файлов в индекс** — нажать на значок `+` (Stage Changes) рядом с нужным файлом.
- **Коммиты** — ввести сообщение в поле *Message* и нажать `Ctrl+Enter` или кнопку *Commit*.
- **Добавление удаленного репозитория** — в терминале выполнить команду `git remote add origin <url>` 
(также можно через интерфейс: *Source Control → ... → Remote → Add Remote*).
- **Получение обновлений** — нажать кнопку *Pull* (значок стрелки вниз).
- **Отправка обновлений** — нажать кнопку *Push* (значок стрелки вверх) или выбрать *Commit & Push* при коммите.
- **Разрешение конфликтов** — при конфликте VS Code подсвечивает проблемные места и предлагает варианты: 
*Accept Current Change*, *Accept Incoming Change*, *Accept Both Changes*, *Compare Changes*. 
После выбора сохранить файл и выполнить коммит.

В ходе предыдущих лабораторных я работала с Git через терминал, а панель *Source Control* использовала только 
для просмотра изменений. Для проверки интеграции в рамках этой работы я внесла тестовое изменение в `README.md`, 
сделала коммит через панель *Source Control* и отправила его на GitHub — всё отработало без ошибок.
