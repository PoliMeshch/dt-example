# Лабораторная работа №5. Средства управления проектами

## Ход работы

### Этап 1. Создание и настройка проекта

#### 1.1. Создание проекта и настройка доступа.
В репозитории GitHub был создан новый проект **Coursework** с типом доски **Kanban** во вкладке **Projects**. 
Преподавателю предоставлен доступ как коллаборатору репозитория.

**Действия:**
1. Открыть репозиторий на GitHub.
2. Перейти во вкладку **Projects**.
3. Нажать **New project**.
4. Выбрать шаблон **Kanban**, назвать проект **Coursework**.
5. Убедиться во вкладке **Settings → Collaborators**, что преподаватель добавлен.

#### 1.2. Настройка этапов для задач.
Настроены этапы (колонки) для реализации заданного рабочего процесса:

- **Backlog** - задача сформулирована, но пока не запланирована к выполнению.
- **Ready** - задача готова к выполнению, исполнитель ещё не приступил.
- **In Progress** - исполнитель работает над задачей.
- **In Review** - работа завершена, задача на согласовании / защите.
- **Back-to-dev** - задача возвращена на доработку.
- **Done** - задача согласована и закрыта.

Рабочий процесс: **Backlog -> Ready -> In Progress -> In Review -> Done**. Если задача не прошла ревью, 
она переходит из **In Review** в **Back-to-dev**, а затем обратно в **In Progress**.

**Действия:**
1. В проекте перейти во вкладку **Settings** (нажать `...` в правом верхнем углу).
2. Выбрать раздел **Status**.
3. Добавить перечисленные этапы.
4. Убедиться, что колонки отображаются на доске в указанном порядке.

#### 1.3. Добавление кастомных полей.
В настройках проекта добавлены поля для маркировки задач:

- **Дата начала** (тип Date) - предполагаемая дата начала работы.
- **Дата окончания** (тип Date) - предполагаемая дата завершения.
- **Тип работы** (тип Single select) - варианты: исследовательская работа, составление документации, программирование.
- **Приоритет** (тип Single select) - варианты: высокий, средний, низкий.
- **Результат** (тип Text) - ожидаемый артефакт.

**Действия:**
1. В проекте нажать `...` (View more options) и выбрать **Settings**.
2. Перейти в раздел **Custom fields**.
3. Добавить перечисленные поля.
4. Сохранить изменения.


### Этап 2. Планирование работ по курсовой работе

#### 2.1. Декомпозиция работы.
Работа над курсовым проектом «Крестики-нолики» разбита на задачи, каждую из которых можно выполнить за 4–10 часов. 
Для каждой задачи указан тип работы и ожидаемый результат.

*Задача 1.* Изучение формулировки задания и составление подробной постановки задачи. Тип: исследовательская работа. 
Результат: текст постановки задачи.

*Задача 2.* Изучение существующих подходов к решению. Тип: исследовательская работа. 
Результат: конспект с источниками.

*Задача 3.* Разработка и согласование алгоритма бота. Тип: исследовательская работа. 
Результат: текстовое описание алгоритма.

*Задача 4.* Реализация алгоритма на C++. Тип: программирование. 
Результат:  код класса игрока.

*Задача 5.* Развертывание проекта. Тип: программирование. 
Результат: собранный проект, пройденные тесты.

*Задача 6.* Изучение требований к написанию отчёта. Тип: составление документации. 
Результат: список требований, структура будущего отчёта.

*Задача 7.* Написание отчёта. Тип: составление документации. Результат: готовый отчёт в формате PDF.

#### 2.2. Создание тикетов.
На основе декомпозиции созданы Issues в репозитории, для каждого заполнены кастомные поля: 
тип работы, приоритет, даты начала и окончания, ожидаемый результат.

**Действия:**
1. Перейти во вкладку **Issues** репозитория.
2. Нажать **New issue**.
3. Указать заголовок задачи.
4. В описании указать, что требуется сделать и какой артефакт должен получиться.
5. Справа в панели проекта выбрать **Coursework** и задать статус.
6. Заполнить кастомные поля.
7. Повторить для всех задач.

#### 2.3. Связи между задачами.
Для задач, результаты которых необходимы для других, установлены блокировки:

- Задача 4 (реализация алгоритма) зависит от задачи 3 (согласование алгоритма).
- Задача 5 (развертывание проекта) зависит от задачи 1 (изучение требований).
- Задача 7 (написание отчёта) зависит от задач 1, 2, 3, 4, 5, 6.

**Действия:**
1. Открыть задачу, которая блокируется.
2. В **Relationships** в поле **Blocked by** выбрать все задачи, от которых она зависит.
3. Проверить, что связь отобразилась в обоих тикетах.
4. Повторить для всех зависимых задач.


### Этап 3. Выполнение задачи
В качестве примера выполнена задача №3 (#11) «Разработка и согласование алгоритма бота».

**Действия:**
1. Создать ветку в репозитории:

```bash
git checkout -b algorithm-description
```

2. Создать файл `reports/coursework/algorithm.md` с описанием алгоритма бота.
3. Сохранить и закоммитить:

```bash
git add reports/coursework/algorithm.md
git commit -m "Первая версия описания алгоритма"
git push origin algorithm-description
```

4. Создать **Draft Pull Request**:
- На GitHub перейти на вкладку **Pull Requests** и нажать **New pull request**.
- Выбрать ветку `algorithm-description` для слияния в `main`.
- В описании PR добавлена строка `Closes #11` для привязки задачи.
- Нажать стрелку рядом с кнопкой **Create pull request** и выбрать **Create draft pull request**.
- В поле **Reviewers** добавить преподавателя.
5. На доске проекта переместить задачу из колонки **Ready** в **In Progress**.
6. На странице **Pull Request** нажать **Ready for review**.
7. Переместить задачу в **In Review**.
8. После ревью и внесения правок перевести задачу в **Done** либо в **Back-to-dev** в зависимости от результата.


### Этап 4. Исследование альтернатив

#### 4.1. Обзор доступных решений.
Рассмотрены популярные системы управления проектами:

- **Jira** - мощная система с Kanban, Scrum, связями задач, кастомными полями, интеграцией с GitHub.
- **Notion** - гибкая среда для документации и ведения задач с настраиваемыми базами данных.
- **Asana** - веб-приложение для управления задачами с удобной визуализацией сроков.
- **YouGile** - российская система, объединяющая таск-трекер и корпоративный мессенджер.

#### 4.2. Анализ альтернативы: YouGile.

**YouGile** - это российская система управления проектами. Главная особенность: каждая задача представляет собой 
отдельный чат, в котором участники могут обсуждать детали, обмениваться файлами и записывать видео с экрана. 
Кроме того, в YouGile заявлен встроенный ИИ-помощник для создания проектов и управления сроками.

**Возможности YouGile:** настройка Kanban-досок, диаграмм Ганта, календарей, кастомные поля и теги, создание 
подзадач и связей через упоминания в комментариях.

**Тарифный план:** бесплатное использование для команд до 10 человек. Если команда больше, стандартный платный 
тариф составляет 495 рублей в месяц за пользователя.

Сценарий выполнения заданий в YouGile:
1. Создать проект и настроить колонки: бэклог, готовы к работе, в работе, на проверке, доработка, готово.
2. Создать карточку задачи, заполнить описание, добавить теги (тип работы), задать дедлайн, назначить ответственного.
3. Приступив к работе, переместить задачу в колонку **В работе**.
4. Результат работы загрузить как вложение или ссылку в чат задачи.
5. Переместить задачу в колонку **На проверке** и упомянуть ревьюера в чате задачи.
6. После утверждения переместить задачу в **Готово**, при необходимости доработки - в **Доработка** и затем обратно в **В работе**.

Отличие от GitHub Projects: в YouGile нельзя напрямую привязать Pull Request, ревью ведётся через обсуждение в чате задачи.

#### 4.3. Вывод по альтернативам.
Для проектов, тесно связанных с кодом и репозиториями, удобнее GitHub Projects, так как он позволяет связывать Issues и 
Pull Requests в одной экосистеме. YouGile больше подходит для команд, которым нужны встроенные коммуникации и которые 
не привязаны к GitHub.
